# 开发文档

## 1. 项目概述

本项目实现了 Andrej Karpathy 提出的"LLM Knowledge Base"范式——一种无需向量检索（RAG）的个人知识库方案。

核心思路：让 LLM 直接阅读全部原始资料，编译为结构化 Markdown Wiki，查询时将整个 Wiki 作为上下文传入 LLM，实现完美召回。

### Karpathy 方案 vs 传统 RAG

| 维度 | 传统 RAG | Karpathy 方案 |
|------|---------|--------------|
| 架构 | 向量DB + Embedding + 重排序 | 纯 Markdown 文件 |
| 召回率 | Top-K 检索，信息丢失 | 完美召回，LLM 看到全部内容 |
| 跨文档关联 | 困难（只能看到检索片段） | 自动发现跨文档概念联系 |
| 基础设施 | 需要向量数据库 | 零基础设施 |
| 适用规模 | 大规模（>100万字） | 个人规模（<50万字） |
| 成本 | Embedding 一次性 + 检索低成本 | 每次查询消耗长上下文 Token |

## 2. 系统架构

六阶段管道设计：

```
Stage 1: Ingest    raw/ 原始资料 → 文本提取
         ↓
Stage 2: Compile   LLM 读取全部文本 → 生成结构化 wiki/
         ↓
Stage 3: Obsidian  使用 Obsidian 打开 wiki/ → 图谱可视化
         ↓
Stage 4: Query     用户提问 → LLM 阅读整个 wiki → 生成答案
         ↓
Stage 5: Output    问答结果 → 保存到 outputs/sessions/
         ↓
Stage 6: Lint      健康检查 → 死链/孤立页面/矛盾检测
```

## 4. 模块详细说明

### 4.1 config.py — 配置管理

从 `.env` 文件加载环境变量，提供全局路径和 API 配置：

- `BASE_DIR` — 项目根目录
- `RAW_DIR` / `WIKI_DIR` / `OUTPUT_DIR` — 各子目录路径
- `OPENAI_API_KEY` / `OPENAI_BASE_URL` / `OPENAI_MODEL` — API 配置
- `SKIP_PATTERNS` — 解析时跳过的目录模式（如 `_files/`）

### 4.2 ingest.py — 文档解析模块

将 raw/ 下的多格式文档统一转换为纯文本。

| 格式 | 解析库 | 函数 |
|------|--------|------|
| .pdf | pdfplumber | `parse_pdf()` |
| .docx | python-docx | `parse_docx()` |
| .pptx | python-pptx | `parse_pptx()` |
| .html/.htm | BeautifulSoup | `parse_html()` |
| .md/.txt | 直接读取 | `parse_markdown()` |

关键函数：
- `ingest_all()` — 遍历 raw/ 解析所有文件，返回 `{文件名: 文本内容}` 字典
- `list_raw_files()` — 列出 raw/ 文件信息（名称、格式、大小、是否可解析）

### 4.3 compiler.py — 知识编译器（核心）

调用 LLM 将原始文本编译为结构化 Wiki 页面。

关键函数：
- `compile_all()` — 全量编译：读取全部 raw → 调用 LLM → 写入 wiki/
- `compile_incremental()` — 增量编译：只处理未被 wiki 引用的新文件
- `_parse_output(text)` — 按 `===FILE: xxx.md===` 分隔符拆分 LLM 输出为多个文件

Compilation Prompt 设计要点：
1. 实体识别：为每个重要概念创建独立页面
2. 自动链接：使用 `[[双向链接]]` 语法
3. Frontmatter：每页包含 title/date/tags/sources
4. 冲突处理：矛盾信息保留并标注
5. 输出格式：`===FILE:===` 分隔，便于程序解析

### 4.4 query_engine.py — 长上下文问答引擎

`KarpathyKnowledgeBase` 类：
- `_load_wiki()` — 加载 wiki/ 全部 .md 拼接为上下文
- `query(question)` — 将完整 wiki 作为 system prompt，调用 LLM 回答
- `save_session()` — 对话记录保存到 outputs/sessions/
- 支持多轮对话，自动维护 `chat_history`

System Prompt 要求 LLM 仅基于知识库回答，并引用具体文件来源。

### 4.5 linter.py — 健康检查

| 检查项 | 函数 | 说明 |
|--------|------|------|
| 死链 | `find_dead_links()` | `[[链接]]` 指向不存在的文件 |
| 孤立页面 | `find_orphan_pages()` | 没被任何页面引用 |
| 空页面 | `find_empty_pages()` | 内容少于 100 字符 |
| Frontmatter | `check_frontmatter()` | 缺少必要字段 |
| 汇总 | `lint_all()` | 执行全部检查 |

### 4.6 exporter.py — 导出模块

- `export_to_html()` — 每个 wiki 页面导出为独立 HTML 文件
- `export_to_single_html()` — 全部页面合并为单个 HTML
- 自动剥离 Frontmatter，转换 Markdown 为 HTML
- 内置 CSS 样式，支持表格、代码块渲染

### 4.7 app.py — 主程序入口

基于 argparse 的命令行界面，使用 rich 库美化输出。

| 子命令 | 功能 | 是否需要 API |
|--------|------|-------------|
| `ingest` | 查看 raw/ 文件解析状态 | 否 |
| `compile` | 全量编译知识库 | 是 |
| `compile --incremental` | 增量编译 | 是 |
| `query "问题"` | 单次问答 | 是 |
| `chat` | 交互式多轮对话 | 是 |
| `lint` | 健康检查 | 否 |
| `export --format html` | 导出 HTML | 否 |

## 6. Wiki 页面规范

每个 wiki 页面遵循以下格式：

```markdown
---
title: "页面标题"
date: 2026-04-28
tags: ["标签1", "标签2"]
sources: ["来源文件1.pdf", "来源文件2.docx"]
---

# 页面标题

## 摘要
简短描述...

## 详细说明
...

## 相关概念
- [[相关页面A]]
- [[相关页面B]]

## 来源
- `来源文件.pdf` — 引用说明
```
